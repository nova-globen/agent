## When to use

Use this when asked to cut, prepare, or ship a release (a new version) of Agent Sync.
Releases are tag-driven: pushing a `v*.*.*` tag runs `.github/workflows/release.yml`, which
builds, tests, publishes the binaries and the NuGet tools, and creates the GitHub Release.
Your job is steps 1–3 below (decide the version, update the solution, commit); the human
runs the push and tag in step 4. `RELEASE_CHECKLIST.md` is the source of truth for the
pipeline and for verifying a release afterward.

## Step 1 — Confirm the next version with the user

Never bump silently. Propose a version, explain why, and wait for confirmation.

1. Find the current version and the latest release:
   - Canonical version: the `<Version>` in `Directory.Build.props` (no leading `v`).
   - Latest tag: `git describe --tags --abbrev=0` (or `git tag --sort=-v:refname | head`).
2. Review what changed since that tag:
   - `git log <latest-tag>..HEAD --oneline`
   - Classify the commits: bug fixes, new features, breaking changes, docs-only.
3. Suggest the next version following the repo's scheme:
   - The line is currently pre-1.0 alpha: `vMAJOR.MINOR.PATCH-alpha.N`. Routine fixes and
     additive features bump the alpha counter (`…-alpha.3` → `…-alpha.4`).
   - Dropping the `-alpha` suffix, or bumping `MINOR`/`MAJOR`, is a bigger call — only
     propose it when the changes justify it, and flag it explicitly.
   - The Git **tag** carries a leading `v` (`v0.2.0-alpha.4`); the `<Version>` property and
     the NuGet package version do **not** (`0.2.0-alpha.4`).
4. State the proposed version and a one-line rationale, then **ask the user to confirm or
   override** before changing any file. Use the confirmed version everywhere below; write
   `<version>` for the bare form and `v<version>` for the tag.

## Step 2 — Update the whole solution to the new version

Do not stop at `Directory.Build.props`. The version and the release narrative appear in
several places; some are generated by `agent sync` and must be changed at their canonical
source, never by hand.

1. **Canonical version** — set `<Version>` in `Directory.Build.props` to `<version>`.
2. **Agent instruction files (generated — edit the skill, then sync):** the version and
   "current status" wording live in the canonical skill
   `.agent/skills/agent-sync-overview/SKILL.md` (its "Status" section). Edit **that** file,
   then run `agent sync` to regenerate `AGENTS.md`, `CLAUDE.md`,
   `.github/copilot-instructions.md`, `.gemini/GEMINI.md`, and the `.claude/skills/` folders.
   **Never hand-edit a generated section** — the markers/lockfile and the drift gate will
   reject it.
3. **Plain docs (edit directly):**
   - `README.md` — any pinned version and feature wording.
   - `RELEASE_CHECKLIST.md` — example tags, if referenced.
   - `.agent/CURRENT_STATE.md` and `.agent/NEXT_STEPS.md` — the "current release" and the
     release-verification items.
   - `.claude/commands/review-release.md` — if it pins a version.
4. **Pinned install versions in the examples** (these gate CI in consuming repos, so keep
   them current):
   - `examples/github-actions/.github/workflows/agent-sync.yml` and its `README.md`
   - `examples/azure-pipelines/azure-pipelines.yml` and its `README.md`
   - They pin `install.sh … | bash -s -- v<version>`; bump the tag.
5. Write the release narrative **honestly** — describe what actually changed since the last
   tag (read the commit log); keep the alpha positioning accurate. Move older, now-historical
   detail into `.agent/CURRENT_STATE.md` rather than bloating the always-loaded files.
6. Find anything missed: `grep -rIn "<previous-version>" --exclude-dir=.git .` (and the
   tag form). Every remaining hit is either intentional history or a spot to update.
7. Verify before committing:
   - `dotnet build --configuration Release` and `dotnet test` are green.
   - `agent status --fail-on-drift --ci` exits `0` (the repo dogfoods Agent Sync — the
     generated files must match the skills). If it reports drift, you hand-edited a
     generated file; fix the canonical skill and re-run `agent sync`.
   - Optionally `scripts/release-smoke.sh` for the publish/UI checks.

## Step 3 — Commit (no AI trailer)

Stage the canonical changes **and** their regenerated projections together and commit. This
repository's convention is **no AI/Claude trailers** in commit messages — do not add a
`Co-Authored-By` / `Claude-Session` / "Generated with" line.

```bash
git add -A
git commit -m "Release v<version>"   # or a short message describing the release
```

Use a clear subject (for example `Release v<version>`); add a brief body summarizing the
notable changes if helpful. Confirm the message contains no AI/Claude references.

## Step 4 — Hand the push and tag back to the user

Do **not** push or create the tag yourself. Print these commands and ask the user to run
them in order (pushing the tag is what triggers the release pipeline):

```bash
git push origin master                # 1. push the release commit
git tag v<version>                     # 2. create the annotated tag
git push origin v<version>             # 3. push the tag -> runs release.yml
```

Then point the user at `RELEASE_CHECKLIST.md` to watch the **Release** (and optional
`release-ui`) workflow finish and verify the GitHub Release assets, `checksums.txt`, the
install scripts, and the `AgentSync` / `AgentSync.Git` / `AgentSync.Ui` NuGet packages.
